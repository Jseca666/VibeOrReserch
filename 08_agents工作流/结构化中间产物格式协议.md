# 结构化中间产物格式协议

本协议规定中间技术产物的机器可读副本格式。目标是降低 Auto-Gate、RD/NXT/EVO/OUT、T2/T5/T4 检索与更新时的 token 消耗，同时保留 Markdown 人类审计轨迹。

本协议只管理格式与读写规则，不替代 `自进化自动工作闭环协议.md`、`产物输出闭环协议.md`、`研究债务队列.md`、`下一轮任务包队列.md`、`产物产出队列.md`、claim 审批或最终对外责任确认。

## 1. 四层格式

| 层级 | 推荐格式 | 主要用途 | 示例 |
|---|---|---|---|
| Markdown 摘要审计层 | `.md` | 人类可读解释、审计记录、裁决理由、阶段小结 | EVO report、NXT/RD/OUT Markdown 队列、研究记忆摘要 |
| CSV/JSONL 机器索引层 | `.csv` / `.jsonl` | 高频大表、单条记录检索、状态更新、跨文件追踪 | raw notes map、定理验证索引、文献索引、active RD/NXT/OUT |
| YAML sidecar 元数据层 | `.yaml` | EVO/产物/任务的快速机器元数据读取 | `EVO_META_TEMPLATE.yaml` |
| LaTeX 数学推导层 | `.tex` | 正式推导、反例笔记、证明包、附录候选 | technical note、derivation log、counterexample note |

Markdown 不再承担所有大表的唯一数据源职责。新增长记录时，优先写结构化副本，再同步 Markdown 摘要或审计说明。

## 1.1 中英分层规则

为降低 token 消耗并避免状态近义词漂移，结构化机器层默认使用英文 key、英文短码和稳定 ID；人类审计层继续使用中文解释。

| 内容类型 | 默认语言 | 说明 |
|---|---|---|
| schema key / CSV 字段 / JSONL 字段 / YAML 字段 | 英文 | 例如 `status`、`privacy_level`、`model_tier`、`worker_model` |
| 状态短码 / 成本等级 / 模型层级 | 英文短码 | 例如 `need_model_card`、`conditional_pass`、`low`、`gate` |
| ID / 路径兼容字段 / 引用键 | 英文或 ASCII 兼容 | 例如 `RN-C011-001`、`PROP-C011-W1-002`、`TLI-C011-001` |
| Auto-Gate 裁决解释 / EVO 正文 / 研究记忆解释 | 中文 | 面向用户和审计者，保持中文可读性 |
| Markdown card 的 YAML front matter | 英文 key + 可选 `*_zh` | 例如 `status: need_model_card` 与 `status_zh: 待模型卡` |

结构化文件中需要同时服务机器与人类时，可使用 `english_key + *_zh` 形式；不得同时冗余写 `label_en` 与英文 key，避免 token 浪费。

新增结构化字段优先采用英文短码；Markdown 摘要可保留中文标题、中文裁决和中文风险说明。

## 2. 隐私字段

所有结构化副本必须包含或继承以下隐私字段：

| 字段 | 取值 | 含义 |
|---|---|---|
| `privacy_level` | `public` / `private` / `sensitive` | 公开边界 |
| `public_allowed` | `true` / `false` | 是否允许进入 GitHub 公开镜像 |
| `redaction_rule` | 文本 | 公开前需要删除或摘要化的内容 |

默认规则：

- 含原始笔记片段摘要、未公开证明细节、技术 note 正文、实验日志、论文正文片段的结构化副本为 `private` 或 `sensitive`。
- 只含公开工作流状态、ID、状态枚举和不敏感摘要的 active CSV 可公开。
- 若同一索引需要公开和私有两版，应拆成 `*_public.csv/jsonl` 与 `*_private.csv/jsonl`。

## 3. 主键与引用规则

| 对象 | 主键字段 | 示例 |
|---|---|---|
| 草稿片段 | `fragment_id` | `RN-C011-001` |
| 候选命题/模型 | `candidate_id` | `PROP-C011-W1-002` |
| 文献索引 | `tli_id` | `TLI-C011-001` |
| 研究债务 | `rd_id` | `RD-20260502-006` |
| 下一轮任务 | `nxt_id` | `NXT-20260503-006` |
| 输出产物 | `out_id` | `OUT-20260503-006` |
| Evolution report | `evo_id` | `EVO-20260503-009` |

CSV 中的多值字段使用分号 `;` 分隔。JSONL 中的多值字段使用数组。所有 ID 必须保持稳定，不能因摘要重写而改变。

## 5. Auto-Gate 默认读取顺序

每轮 Auto-Gate 应优先读取小型结构化副本，再按需读取 Markdown/LaTeX 正文：

```text
1. contribution_evidence_matrix 或其结构化副本；
2. raw_notes_anchor_map.csv 中相关 fragment_id；
3. raw_notes_theorem_validation_index.jsonl 中相关 candidate_id；
4. theorem_literature_index.jsonl 中相关 tli_id；
5. research_debts_active.csv / nxt_queue_active.csv / out_queue_active.csv；
6. 必要时读取 Markdown 审计说明、LaTeX 推导正文、实验日志或原始笔记。
```

若结构化副本与 Markdown 摘要冲突，Auto-Gate 必须：

```text
自动暂缓；
登记或更新 RD/NXT；
生成一致性修复任务；
不得升级 claim 或关闭 claim阻断债务。
```

## 6. 写回规则

- 新增草稿片段、候选命题、文献 grounding、active RD/NXT/OUT 时，优先写结构化副本。
- Markdown 队列继续保留人类审计入口，只同步摘要、裁决、边界和下一步。
- 结构化副本不替代 claim 审批、证明审计、文献精读、实验验证或最终对外责任确认。
- `EVO report` 继续使用 Markdown；若涉及多项 RD/NXT/OUT 和记忆写回，应附带 `.meta.yaml` sidecar。
- 任何结构化副本中的 `status` 只表示系统内状态，不表示论文强 claim 可写。
- 涉及模型路由或并行调度的结构化记录，应同步写入 `model_tier`、`worker_model`、`review_model`、`parallelizable`、`merge_owner`、`escalation_reason`、`cost_class`；字段含义以 `model_routing_policy.md` 和 `并行agents调度协议.md` 为准。

## 7. 状态枚举建议

### 草稿片段论文角色

```text
主贡献候选 / 方法接口 / 附录技术基础 / 限制讨论 / 背景 / 暂缓 / 内部技术债务 / 剔除当前主线
```

### 验证任务状态

```text
无需验证 / 待T2 / 待T5 / 待T4 / 已索引 / 待模型卡 / 待证明审计 / 暂缓 / 剔除当前主线
```

### 候选命题状态

```text
仅有推导 / 待假设账本 / 待证明审计 / 待反例搜索 / 待文献索引 / 待模型卡 / 系统内条件成立候选 / 被反例击穿 / 文献已有 / 自动降级 / 暂缓 / 剔除当前主线
```

### 文献关系

```text
直接已有 / 部分支撑 / 条件不同 / 模型相近 / 空白缺口 / 背景引用 / 审稿风险
```

## 8. 最小迁移顺序

```text
1. 新增本协议和结构化模板，不删除旧 Markdown。
2. 为 raw_notes_anchor_map 生成私有 CSV 副本。
3. 为 theorem validation / literature index 生成 JSONL 副本。
4. 为 RD/NXT/OUT 生成 active CSV 摘要。
5. 为 EVO 增加 meta.yaml sidecar 模板。
6. 后续 Auto-Gate 默认先读结构化副本，再按需读取 Markdown/LaTeX 正文。
```

## 9. 禁止事项

- 不得把结构化副本中的 `系统内条件成立候选` 写成论文定理已证。
- 不得因为某条 CSV/JSONL 状态为 `已索引` 就关闭 RD 或升级 claim。
- 不得把私有 raw notes、未公开证明、实验日志或论文正文片段推送到公开镜像。
- 不得用结构化副本替代人类责任确认、投稿前确认或公开发布前确认。
