# Pro 外部评审对接流程

本文件记录与网页版 GPT Pro 对接时可采用的一种 agents 辅助调度方式。它主要供本地 Codex/agents 遵守，不是论文工作调度主链、论文内容、研究贡献或必须对外展示的项目结构。目标是让外部评审能读取足够材料、给出针对性建议，同时把可公开的通用工作流稳定同步到 GitHub 固定 commit，并避免把未公开论文内容放入公开仓库。

## 1. 固定流程

1. 明确本轮 Pro 评审目标：工作流审查、证明核验、文献边界、模型可解性、实验设计、论文叙事、工具利用或用户可用性。
2. 选择本轮审计模式：`公开镜像优先`、`本地核验优先` 或 `混合审计`。
3. 区分两类材料：GitHub 公开镜像材料与本地临时 Pro 核验材料。二者的安全边界不同：GitHub 面向开源社区，必须保守；本地 handoff 包只上传到私密 Pro 对话 source，可按审计目标放入必要的原始笔记、证明细节、局部论文草稿或实验核验材料。
4. 若本轮审查完整工作流、角色分工、Auto-Gate、RD/NXT/EVO/OUT、研究记忆、paper_strategy 或产物规范等通用链路，默认更新 GitHub 公开镜像，只同步可公开内容，并记录固定 commit。
5. 若本轮只审查 `01_原始笔记/`、未公开证明细节、论文正文草稿、实验原始数据或其他敏感核验材料，可跳过 GitHub 更新，但必须在 `README_上传说明.md` 中说明“不更新 GitHub”的原因。
6. 在 `C:\Users\dzw\Desktop\research\_pro_handoff_tmp\` 下新建本轮 handoff 子目录。
7. 将本轮必要核验文件复制到 handoff 子目录；若本轮需要 Pro 实质核验数学、文献、证明、脚本或论文片段，临时包应给足必要材料，不要因 GitHub 公开限制而过度删减。若材料超过外层上传项限制，先复制到临时材料子目录并打包为 zip。
8. 生成 `PROMPT_给Pro.md` 和 `README_上传说明.md`；二者必须写明审计模式、固定 commit、GitHub 用途、本地 zip 用途和禁止事项。
9. 上传前清理 zip 打包前的临时材料子目录，handoff 外层只保留实际要上传的文件，例如 `PROMPT_给Pro.md`、`README_上传说明.md` 和一个或多个 zip。
10. 用户把 handoff 子目录中的外层上传项上传给 Pro，并要求 Pro 读取 `PROMPT_给Pro.md`。
11. 回收 Pro 反馈后，由 Codex 审查其建议是否能进入项目门禁。
12. 若建议影响 claim、研究债务、角色职责、任务包、evolution report、OUT 产物、研究记忆或 paper_strategy，写回对应项目文件。

## 2. 审计模式

### 2.1 公开镜像优先

适用于完整工作流和通用链路审查。默认用于：

- AGENTS、Auto-Gate、T0-T7 调度、角色分工、门禁协议。
- RD/NXT/EVO/OUT 队列、产物输出规范、Pro 对接规则。
- `07_研究记忆/` 中可公开的方向记忆、证明策略记忆和实验记忆摘要。
- `04_论文草稿/paper_strategy/` 中不含正文细节的论文主题、贡献证据矩阵、related work map 和审稿风险。

执行要求：

- 更新 GitHub 公开镜像，只 stage 可公开材料。
- 提交并 push 后记录固定 commit。
- `PROMPT_给Pro.md` 和 `README_上传说明.md` 必须把固定 commit 放在显眼位置，并要求 Pro 以固定 commit 为主审对象，避免读取 GitHub 默认页面缓存。
- 本地 zip 可作为补充核验材料，不替代固定 commit。若本轮公开链路审计还需要核验具体笔记、证明、脚本或局部论文片段，应在本地 zip 中提供这些私有 source，并明确其只作核验，不进入公开镜像。

### 2.2 本地核验优先

适用于只涉及私有或未公开材料的审查。默认用于：

- 完整 `01_原始笔记/`、代表性原始草稿片段。
- 未公开证明细节、证明草稿、局部论文正文、审计草稿。
- 实验原始数据、求解器详细日志、尚未脱敏结果表。

执行要求：

- 默认不更新 GitHub。
- `README_上传说明.md` 必须写明不更新 GitHub 的原因。
- 本地 zip 必须标注哪些材料只作核验，不代表已验证结论，不得用于 claim 升级。

### 2.3 混合审计

适用于同时需要公开工作流和私有核验材料的审查。常见于：

- 让 Pro 同时审查通用工作流和少量原始草稿片段。
- 让 Pro 以公开 RD/NXT/EVO/OUT 为主线，再用本地 proof/source 材料核验某个环节。
- 用户视角压力测试中，需要公开入口规则，也需要少量私有草稿片段模拟输入。

执行要求：

- GitHub 固定 commit 提供公开主线。
- 本地 zip 放无法公开或不宜公开的补充核验材料，包括必要的原始笔记、证明细节、局部论文草稿、实验脚本结果或目录清单。
- `PROMPT_给Pro.md` 必须明确：公开 commit 是正式工作流状态，本地 zip 是补充核验材料；Pro 反馈不直接改变项目状态。

## 3. GitHub 公开镜像材料

公开镜像是通用工作流审计的默认主入口，不是敏感材料核验的承载区。凡是审查完整工作流、角色分工、Auto-Gate、RD/NXT/EVO/OUT、研究记忆、paper_strategy、产物规范或用户入口，默认使用公开镜像优先模式。公开镜像只放可公开的工作流与索引材料，例如：

- `AGENTS.md`
- `00_总览索引/` 中的项目地图、claim 状态、任务看板、符号表和公开摘要。
- `03_文献综述/` 中的文献分类、文献边界模板或非敏感综述材料。
- `04_论文草稿/paper_strategy/` 中不含正文细节的论文主题、贡献候选、related work map、贡献证据矩阵和审稿风险。
- `07_研究记忆/` 中可公开的方向记忆、证明策略记忆和实验记忆摘要。
- `08_agents工作流/` 中的角色、门禁、产物规范、研究债务、任务包和 evolution report。

公开镜像不得放：

- 完整 `01_原始笔记/`。
- 完整论文正文、未公开证明细节或未公开实验原始数据。
- 私有数据、账号、token、私钥、环境变量或其他敏感信息。
- 与本轮评审无关的全量材料。

公开镜像同步前必须检查 staging 范围。若工作区中存在 `01_原始笔记/`、未公开证明或实验细节的改动，必须只 stage 明确可公开文件，不能使用模糊的全量 add。

## 4. 本地临时 Pro 核验材料

本地临时 handoff 目录是私有核验材料的主要入口，可以按本轮任务必要性组织“必要全量核验包”。这里的“必要全量”不同于 GitHub 公开镜像：只要服务本轮 Pro 审计目标，就可以给 Pro 足够的 source 材料进行实质核验。它可以包含：

- 命题证明细节、证明草稿、审计草稿。
- 公式推导片段、代表性原始笔记、局部论文草稿。
- 文献边界材料、实验摘要、求解器日志摘要或局部结果表。
- 本轮 Pro 需要交叉核验的公开镜像链接和固定 commit。

临时目录不是正式项目结构，不作为最终状态来源。正式状态仍以项目内 `研究债务队列.md`、`下一轮任务包队列.md`、`evolution_reports/`、`claim审批表.md` 和命题目录为准。

原则：不得把 GitHub 的公开限制误套到本地 Pro 包上。GitHub 禁止放完整原始笔记、未公开证明、论文正文和敏感实验数据；本地 Pro 包可以在私密对话中放入这些材料的必要部分或必要全量版本，前提是本轮评审确实需要，并且 `README_上传说明.md` 明确其只作核验、不代表已验证结论、不得直接用于 claim 升级或债务关闭。

## 5. 文件数量规则

- 默认控制在 20 个上传项以内，以适配 Pro 单对话文件限制。
- 若本轮材料超过 20 个文件，优先把核验材料打包为一个或多个 zip，并在外层保留 `PROMPT_给Pro.md` 与 `README_上传说明.md`。
- zip 打包完成后，handoff 外层不得继续保留被打包前的未压缩材料目录；否则用户上传整个目录时会把 zip 和未压缩材料一并上传，导致超过文件数量限制。
- handoff 外层只允许保留实际上传项：`PROMPT_给Pro.md`、`README_上传说明.md`、zip 包，以及必要时少量必须单独上传的补充文件。
- 每个 handoff 子目录默认控制在 40 个项目以内；若超过 40 个项目，应使用 zip 包或拆分为多轮 Pro 对接包。
- 单个 zip 内可以保留相对目录层级，并放入 `MANIFEST_文件清单.md` 说明文件用途。
- 若 Pro 无法读取 zip，则由用户解压后按 `README_上传说明.md` 中的优先级拆成多轮上传。

文件数量多时，必须在 `README_上传说明.md` 中列出文件清单或分组，避免 Pro 把核验材料误读为全部项目状态。

上传前必须执行外层目录检查：若存在 zip 对应的未压缩材料目录，应先删除该临时目录，只保留外层上传项。

## 6. 固定文件

每个 handoff 子目录至少包含：

- `PROMPT_给Pro.md`：Pro 的主入口，写明任务、固定 commit、上传文件用途、输出结构和禁止事项。
- `README_上传说明.md`：说明本轮目录如何使用、哪些文件来自公开镜像、哪些文件只作临时核验。

`PROMPT_给Pro.md` 应明确：

- 本轮评审目标。
- 本轮审计模式：`公开镜像优先`、`本地核验优先` 或 `混合审计`。
- 若使用 GitHub，固定 commit 是什么，Pro 应优先读取哪些链接。
- 若使用本地 zip，本地 zip 与 GitHub commit 的分工是什么。
- 是否允许使用临时 proof/source 材料核验。
- 哪些结论不得直接升级 claim。
- 输出结构和裁决格式：`通过 / 有条件通过 / 不通过`。
- 若使用 GitHub 链接，优先使用固定 commit，避免读取缓存旧版本。

`README_上传说明.md` 应明确：

- 本轮是否更新 GitHub；若不更新，原因是什么。
- 固定 commit 与本地 zip 的优先级。
- 上传文件清单和用途。
- 哪些文件只作临时核验，不代表项目正式状态。
- 若本地 zip 包含原始笔记、证明细节、论文片段或实验材料，必须说明这些材料可供 Pro 核验，但不得视为公开镜像内容，也不得直接升级 claim、关闭 RD 或进入成果型输出。
- Pro 反馈应如何被 Auto-Gate 吸收，不能直接升级 claim 或关闭债务。

## 7. 多轮工作流复审默认安排

若用户要求 Pro 复审当前数学与运筹学 agents 工作流是否已经形成自适应科研闭环，默认拆成多轮专项审计：

| 轮次 | 审计主题 | 默认模式 | 主要材料 |
|---|---|---|---|
| 第 1 轮 | 闭环逻辑审计 | 公开镜像优先；若需核验真实执行链可附本地 source | AGENTS、T0-T7、Auto-Gate、RD/NXT/EVO/OUT、paper_strategy、研究记忆；必要时附代表性原始输入或执行证据 |
| 第 2 轮 | 工作细节完备性审计 | 公开镜像优先；必要时混合审计 | 模板、字段、状态、触发条件、队列、report、产物准入；必要时附代表性 report/source |
| 第 3 轮 | 工具与 EvoScientist 利用审计 | 公开镜像优先；涉及本地环境日志时混合审计 | EvoScientist 对齐、研究记忆、技能路由、工具申请规则、实验记忆；必要时附本地环境日志 |
| 第 4 轮 | 角色分工与冗余审计 | 公开镜像优先 | 角色职责矩阵、Codex 调用协议、Auto-Gate、角色变更记录、典型任务链 |
| 第 5 轮 | 基础薄弱用户压力测试 | 混合审计 | AGENTS、README、Auto-Gate 入口、NXT-20260503-002、paper_strategy、少量私有草稿目录清单或代表性片段 |

多轮复审的每轮反馈都必须由 Auto-Gate 吸收。若 Pro 建议公开镜像更新后的工作流再复审，应优先生成新的固定 commit，而不是让 Pro 读取 GitHub 默认页面。

## 8. 安全边界

临时目录可以包含论文证明细节，但必须服务本轮评审目标。临时目录也不得放：

- token、账号、私钥、API key、cookie 或环境变量。
- 未脱敏隐私数据、敏感实验数据或无关全量资料。
- 与本轮问题无关的大量原始文件。

GitHub 公开镜像与临时 Pro 包必须分层管理：公开镜像承载通用工作流和可公开索引，临时目录承载私有核验材料。不得为了方便 Pro 审查而把完整原始笔记、未公开证明、论文正文或敏感实验数据推送到公开仓库；也不得反过来因为 GitHub 不能放这些材料，就让本地 Pro 核验包缺少本轮审计所需的 source。

## 9. Pro 反馈吸收规则

Pro 反馈是外部审查建议，不替代：

- Auto-Gate 门禁裁决或最终对外责任确认。
- claim 审批表。
- 研究债务关闭。
- 角色变更落档。
- 证明审计、文献边界审查或实验验证。

若 Pro 反馈影响项目状态，必须写回：

- `08_agents工作流/研究债务队列.md`
- `08_agents工作流/下一轮任务包队列.md`
- `08_agents工作流/evolution_reports/`
- `08_agents工作流/角色变更记录.md`
- `00_总览索引/claim审批表.md` 或对应命题目录

未写回前，Pro 反馈只视为对话建议，不视为项目已采纳结论。
