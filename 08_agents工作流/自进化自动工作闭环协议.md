# 自进化自动工作闭环协议

本文件规定一次研究输入如何进入 T1-T7、如何调用角色、如何验收、如何登记债务、如何写回记忆、如何触发角色自进化、如何生成 evolution report 和下一轮任务包。

本文件是轻量闭环控制层，不重写 `Codex角色调用协议.md`、`角色职责矩阵.md`、`研究债务队列.md`、`角色变更记录.md` 和 `产物输出格式规范.md`。若发生冲突，以 `AGENTS.md`、`角色职责矩阵.md` 和 `研究债务队列.md` 的门禁与裁决规则为准。

## 1. 适用范围

任一研究任务在开始前先按本协议判断是否需要启动一轮闭环。闭环目标不是替代技术专家、EvoSkills 或 `PI导师`，而是确保每轮任务有明确入口、产物、验收、债务、记忆和下一步。

本协议适用于：

- 公式、命题、模型、实验、文献、论文草稿、claim 审批和角色自进化任务。
- 当前优先 claim、未来新 claim、新理论方向、新实验任务和新反例任务。
- 轻量任务和关键任务；关键任务仍必须按 `Codex角色调用协议.md` 使用关键任务调用包。

本协议不适用于：

- 直接裁决 claim 强度。
- 关闭 `claim阻断` 债务。
- 修改角色结构或主线方向。
- 将未审查推导、猜想或一次性实验异常写成稳定结论。

以上事项必须由 `PI导师` 或对应门禁角色按既有规则裁决。

## 2. 输入触发源

出现以下任一输入时，必须启动一轮闭环判断：

- 新公式、新推导、新定义或符号冲突。
- 新模型、新 reformulation、新 MILP 嵌入或算法设想。
- 新文献、文献边界变化、related work 草稿或新颖性风险。
- 新实验结果、实验异常、求解器日志、benchmark 变化或反例。
- 未关闭研究债务的状态更新、逾期、暂缓或关闭请求。
- claim 状态变化请求，或论文草稿中新增、强化贡献表述。
- 角色职责不匹配、同类交付物反复退回、任务长期卡住。
- `PI导师` 裁决、修改意见或主线取舍要求。

如果输入不需要展开任务，也必须给出轻量判断，例如“无需新增闭环；不产生债务；不进入 claim 升级”。

## 3. T1-T7 自动分类记录

每轮任务必须先填写分类记录，再进入角色调用。

```text
主任务类型：
辅助任务类型：无 / T1-T7
分类理由：
是否关键任务：是 / 否
是否关联 claim：无 / claim 编号 / 待定
是否可能产生新 claim：是 / 否 / 待判断
是否触发 T7：是 / 否 / 待判断
```

分类规则：

- 公式、符号、映射和推导链整理优先归入 T1。
- 定义、引理、命题、定理和证明推进优先归入 T2。
- MILP、reformulation 嵌入、变量约束、可解性和算法结构优先归入 T3。
- 反例、小规模验证、benchmark、求解器日志和 out-of-sample 检查优先归入 T4。
- 文献分类、边界、新颖性和 related work 风险优先归入 T5。
- claim 状态升级、论文强 claim、贡献表述写入优先归入 T6。
- 角色职责不匹配、职责调整、主线调整或跨周期经验抽象优先归入 T7。

同一轮可以多标签，但必须指定一个主任务类型。若分类理由不清，应登记为研究债务或在 evolution report 中标为 `待查`。

## 4. 闭环运行步骤

每轮任务按以下顺序运行：

```text
输入触发
  -> T1-T7 分类
  -> 生成轻量或关键任务调用包
  -> 调用角色和必要 EvoSkills
  -> 产物按角色验收口径检查
  -> 更新研究债务
  -> 判断研究记忆写回
  -> 判断是否触发 T7
  -> 汇总 PI待裁决事项
  -> 生成 evolution report
  -> 生成下一轮任务包或说明暂无下一轮
```

闭环不得在以下事项缺失时收口：

- 本轮产物是否通过验收。
- 新增或更新债务是否进入 `研究债务队列.md`。
- 未关闭 `阻塞` 和 `claim阻断` 债务是否列明。
- 是否需要写回 `07_研究记忆/`。
- 是否需要触发 T7 角色或主线自进化。
- 是否存在 `PI导师` 待裁决事项。
- 是否生成下一轮任务包，或明确“暂无下一轮任务包”。

## 5. 研究债务驱动下一轮

研究债务不仅是记录，也是下一轮任务包的重要来源。

- `普通` 债务必须持续跟踪，但不阻断本轮继续推进。
- `阻塞` 债务必须生成下一轮处理动作，或说明暂缓理由和负责角色。
- `claim阻断` 债务必须生成下一轮处理动作，并列入 `PI导师` 待裁决事项；未关闭前不得升级论文强 claim。
- 如果债务暂缓，必须说明暂缓原因、重新触发条件和是否需要 `PI导师` 裁决。

`研究债务协调员` 可以汇总、催办和生成下一轮任务包建议，但不能自行关闭 `阻塞` 或 `claim阻断` 债务。

## 6. 研究记忆写回规则

研究记忆只写入跨周期稳定经验，不替代研究债务、claim 审批、文献边界表或 PI 裁决。

可以写回：

- 可复用证明策略、失败证明路线和常见缺口。
- 可复用实验配置、求解器经验、异常排查经验和复现实验线索。
- 文献边界经验、已有结果边界和 related work 风险模式。
- 方向取舍经验、暂缓方向、失败方向和重新激活条件。

不得写回为稳定结论：

- 未审查 claim。
- 未关闭研究债务。
- 猜想、直觉解释或一次性实验异常。
- 尚未完成文献边界审查的新颖性判断。
- 未经 `PI导师` 裁决的主线变化。

写回建议应在 evolution report 中列明写入位置：`方向记忆.md`、`证明策略记忆.md`、`实验记忆.md` 或“无”。

## 7. T7 触发规则

出现以下任一情况时，必须判断是否触发 T7：

- 连续两轮出现同类角色缺口。
- 同一角色交付物连续两轮被退回。
- 债务类型为 `待更新角色`。
- 文献边界、证明缺口或实验异常反复阻断同一类任务。
- 当前角色组合无法覆盖新增任务类型或外部输入要求。
- `PI导师` 要求调整主线、弱化 claim、暂停方向或新增角色。

触发 T7 不等于角色结构自动变化。角色变更必须写入 `角色变更记录.md`，并由 `PI导师` 裁决后才能落档。

## 8. Evolution Report 模板

```markdown
# Evolution Report

| 字段 | 内容 |
|---|---|
| 轮次 ID | EVO-YYYYMMDD-序号 |
| 日期 | YYYY-MM-DD |
| 输入触发 | 新公式 / 新文献 / 新实验 / 反例 / 债务更新 / PI意见 / 论文草稿 / 角色冲突 / 其他 |
| 主任务类型 | T1-T7 |
| 辅助任务类型 | 无 / T1-T7 |
| 分类理由 | 为什么判为该任务类型 |
| 关联 claim | 无 / claim 编号 / 待定 |
| 调用角色 | 角色列表 |
| 使用 EvoSkills | 无 / 技能列表 |
| 本轮产物 | 文件、表格、报告或结论边界 |
| 验收结论 | 通过 / 有条件通过 / 不通过 / 待查 |
| 新增或更新债务 | 无 / RD-YYYYMMDD-序号 |
| 未关闭阻塞债务 | 无 / RD-YYYYMMDD-序号 |
| 未关闭 claim阻断债务 | 无 / RD-YYYYMMDD-序号 |
| 记忆写回 | 无 / 方向记忆 / 证明策略记忆 / 实验记忆 |
| 是否触发 T7 | 是 / 否 |
| 角色变更建议 | 无 / RCP-YYYYMMDD-序号 |
| PI待裁决事项 | 无 / 事项列表 |
| 下一轮任务包 | 无 / NXT-YYYYMMDD-序号 |
```

每轮任务结束必须生成 evolution report，或明确说明“本轮为轻量问答，不产生 report；无新增债务；无下一轮任务包”。涉及关键任务时不得省略 report。

关键任务的 evolution report 必须归档到 `evolution_reports/`，命名与 `轮次 ID` 保持一致。evolution report 只记录本轮闭环状态，不替代 claim 审批、债务关闭、角色变更或 `PI导师` 裁决。

## 9. 下一轮任务包模板

```markdown
# 下一轮任务包

| 字段 | 内容 |
|---|---|
| 任务包 ID | NXT-YYYYMMDD-序号 |
| 来源 | 债务 / 角色变更 / 记忆写回 / PI裁决 / 文献边界 / 实验异常 / 反例 / 论文草稿 |
| 建议任务类型 | T1-T7 |
| 目标产物 | 下一轮必须交付的文件、表格或报告 |
| 必须调用角色 | 角色列表 |
| 可选 EvoSkills | 无 / 技能列表 |
| 阻断条件 | 无 / 阻塞债务 / claim阻断债务 / 文献边界未定 / 实验设计未定 |
| 进入 PI 裁决条件 | 无 / 条件列表 |
| 暂缓条件 | 无 / 暂缓原因与重新触发条件 |
```

下一轮任务包可以由 `研究债务协调员` 汇总建议，但技术内容必须交给对应角色处理；claim、主线和角色结构仍由 `PI导师` 裁决。

生成的下一轮任务包必须同步登记到 `下一轮任务包队列.md`；若无下一轮任务包，必须在 evolution report 中写明“暂无下一轮任务包”。

## 10. 最小运行规则

- 每轮必须分类：至少说明主任务类型、分类理由和是否关键任务。
- 每轮必须收口：至少说明产物、验收结论、债务影响和 PI待裁决事项。
- 每轮必须生成下一轮任务包，或明确“暂无下一轮任务包”。
- 关键任务必须生成 evolution report。
- 关键任务的 evolution report 必须归档，下一轮任务包必须进入队列。
- C-010/C-011 只是当前优先 claim 示例，不是闭环入口；新 claim、新理论方向、新实验任务和新文献边界任务均按 T1-T7 进入闭环。
