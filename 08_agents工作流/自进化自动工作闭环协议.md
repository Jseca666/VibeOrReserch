# 自进化自动工作闭环协议

本文件规定一次研究输入如何进入 T0-T7、如何调用角色、如何验收、如何登记债务、如何写回记忆、如何触发角色自进化、如何生成 evolution report 和下一轮任务包。

本文件是轻量闭环控制层，不重写 `Codex角色调用协议.md`、`角色职责矩阵.md`、`研究债务队列.md`、`角色变更记录.md`、`产物输出格式规范.md` 和 `产物输出闭环协议.md`。若发生冲突，以 `AGENTS.md`、`角色职责矩阵.md` 和 `研究债务队列.md` 的门禁与裁决规则为准。

## 1. 适用范围

任一研究任务在开始前先按本协议判断是否需要启动一轮闭环。闭环目标不是替代技术专家或 EvoSkills，而是通过 Auto-Gate 自动门禁裁决层确保每轮任务有明确入口、产物、验收、债务、记忆和下一步。

本协议适用于：

- 公式、命题、模型、实验、文献、论文草稿、claim 审批和角色自进化任务。
- 当前优先 claim、未来新 claim、新理论方向、新实验任务和新反例任务。
- 轻量任务和关键任务；关键任务仍必须按 `Codex角色调用协议.md` 使用关键任务调用包。

本协议不适用于最终对外责任确认，包括：

- 投稿、公开发布或最终论文强 claim 表述。
- 静默安装工具、求解器、系统组件或外部许可证。
- 代表用户作出算力投入、方向偏好或重大主线取舍。
- 将未审查推导、猜想或一次性实验异常写成稳定结论。

日常 claim 状态候选、债务推进、暂缓、降级、关闭候选和下一轮任务生成，由 Auto-Gate 按保守规则进行系统内裁决；系统内裁决不等同于最终对外责任确认。

## 2. 输入触发源

出现以下任一输入时，必须启动一轮闭环判断：

- 新公式、新推导、新定义或符号冲突。
- 新模型、新 reformulation、新 MILP 嵌入或算法设想。
- 新文献、文献边界变化、related work 草稿或新颖性风险。
- 论文主题、贡献策略、related work map、贡献证据矩阵或审稿风险发生变化。
- 新实验结果、实验异常、求解器日志、benchmark 变化或反例。
- 未关闭研究债务的状态更新、逾期、暂缓或关闭请求。
- claim 状态变化请求，或论文草稿中新增、强化贡献表述。
- 角色职责不匹配、同类交付物反复退回、任务长期卡住。
- Auto-Gate 裁决、用户偏好确认、对外责任确认或主线取舍要求。

如果输入不需要展开任务，也必须给出轻量判断，例如“无需新增闭环；不产生债务；不进入 claim 升级”。

## 3. T0-T7 自动分类记录

每轮任务必须先填写分类记录，再进入角色调用。

```text
主任务类型：
辅助任务类型：无 / T0-T7
分类理由：
论文锚点 / 贡献候选 / 证据缺口：
是否关键任务：是 / 否
是否关联 claim：无 / claim 编号 / 待定
是否可能产生新 claim：是 / 否 / 待判断
是否触发 T7：是 / 否 / 待判断
```

分类规则：

- 论文主题、贡献假设、related work 主线、贡献证据矩阵和审稿风险优先归入 T0。
- 公式、符号、映射和推导链整理优先归入 T1。
- 定义、引理、命题、定理和证明推进优先归入 T2。
- MILP、reformulation 嵌入、变量约束、可解性和算法结构优先归入 T3。
- 反例、小规模验证、benchmark、求解器日志和 out-of-sample 检查优先归入 T4。
- 文献分类、边界、新颖性和 related work 风险优先归入 T5。
- claim 状态升级、论文强 claim、贡献表述写入优先归入 T6。
- 角色职责不匹配、职责调整、主线调整或跨周期经验抽象优先归入 T7。

同一轮可以多标签，但必须指定一个主任务类型。若分类理由不清，应登记为研究债务或在 evolution report 中标为 `待查`。

### 3.1 论文锚点检查

每个重技术任务在进入 T1-T7 前，必须完成论文锚点检查：

```text
论文主题：
贡献候选：
证据缺口：
related work 缺口：
是否只是内部技术债务：
是否需要先触发 T0：
```

若一个技术 NXT 无法说明服务的论文主题、贡献候选或证据缺口，Auto-Gate 默认自动暂缓技术扩展，并生成 T0 论文方向与贡献策略任务。若该任务只是补证据、修反例或整理旧材料，可标为内部技术债务继续推进，但不得进入 proof package、paper progress、slides 或成果型输出。

### 3.2 定理验证与文献索引触发规则

T0 论文锚点分流只回答“草稿片段是否有论文用途”。凡是片段进入主贡献候选、方法接口候选、附录技术基础、定理候选或优化改革写候选，还必须继续判断是否触发 T2 定理推导验证、T5 文献索引与边界检查，以及必要的 T4 反例/小规模验证。

定理验证触发规则：

```text
凡是草稿片段包含：
- 定理 / 引理 / 命题；
- 等价改革写；
- 最坏概率对偶；
- MILP 精确性；
- 有限枚举上界；
- 支撑等价；
- 推前等价；
- 可解性或算法正确性 claim；

必须触发 T2 定理推导验证。
若没有 T2 验证记录，只能保持：
仅有推导 / 待审计 / 建模假设 / 暂缓。
```

文献索引触发规则：

```text
凡是草稿片段被标为：
- 主贡献候选；
- 方法接口候选；
- 附录技术基础；
- 定理候选；
- 优化改革写候选；
- 模型/算法 claim 候选；

必须触发 T5 文献索引检查。
若没有 T5 文献索引，不得进入：
proof package / paper progress / main.tex / 成果型 slides / claim 升级。
```

验证闭环最小字段：

```text
候选 ID：
来源片段：
候选类型：
假设账本：
证明步骤矩阵：
反例入口：
文献索引：
当前状态：
关联 RD/NXT/OUT：
Auto-Gate 裁决：
```

SRI/CVaR/SOC、普通能耗 Wasserstein 球、早期风险度量、SOC/cuts 等非当前 DRCC-MILP 主线片段，默认标为 `剔除当前主线`。它们只保留原始笔记归档，不触发本轮 T2/T4/T5，不生成新 RD/NXT/OUT；除非未来 T0 明确重启该分支。

## 4. 闭环运行步骤

每轮任务按以下顺序运行：

```text
输入触发
  -> 论文锚点检查
  -> T0-T7 分类
  -> 判断定理验证 / 文献索引 / 反例验证触发
  -> 生成轻量或关键任务调用包
  -> 调用角色和必要 EvoSkills
  -> 产物按角色验收口径检查
  -> 更新研究债务
  -> 判断研究记忆写回
  -> 判断是否触发正式输出产物
  -> 判断是否触发 T7
  -> 执行 Auto-Gate 自动门禁裁决
  -> 标记是否需要人类偏好确认或对外责任确认
  -> 生成 evolution report
  -> 生成下一轮任务包或说明暂无下一轮
```

### 4.1 工具缺失与安装申请规则

若任务执行中发现缺少解释器、求解器、包管理器、系统组件、Python/R/Julia/LaTeX/Sage 环境或其他外部工具，必须显式提出工具配置申请，不得只把缺口藏在任务报告、研究债务或下一轮任务包中。

工具配置申请至少包含：

- 缺失工具名称、建议版本和当前检测结果。
- 该工具服务的任务、关联 NXT/RD/EVO，以及是否阻塞当前任务。
- 建议配置方式：使用用户已有路径、安装新工具、暂缓执行或改用替代方案。
- 是否会改变系统环境、项目环境或依赖状态。
- 若后续需要额外包、求解器或许可证，必须单独申请。

在用户明确确认前，不得安装、升级或配置新获取的软件，也不得把“准备安装”写成任务已完成。可以继续推进不依赖该工具的文档整理、协议设计、脚本草案和风险登记；需要真实运行的部分应保持 `进行中` 或 `暂缓`，并在 RD/NXT/EVO 中标注为 `待用户确认工具配置`。

闭环不得在以下事项缺失时收口：

- 本轮产物是否通过验收。
- 重技术任务是否完成论文锚点检查，或明确标为内部技术债务。
- 定理型、改革写型或模型型片段是否完成 T2/T5/T4 触发判断，或明确标为 `剔除当前主线` / `暂缓`。
- 新增或更新债务是否进入 `研究债务队列.md`。
- 未关闭 `阻塞` 和 `claim阻断` 债务是否列明。
- 是否需要写回 `07_研究记忆/`。
- 是否触发 `产物产出队列.md` 中的 OUT 产物，或明确“暂无输出产物触发”。
- 是否需要触发 T7 角色或主线自进化。
- Auto-Gate 自动裁决状态是否列明。
- 是否存在人类偏好确认或对外责任确认需求。
- 是否生成下一轮任务包，或明确“暂无下一轮任务包”。

## 5. Auto-Gate 自动门禁裁决

Auto-Gate 是系统内保守裁决层，负责日常科研推进中的路由、验收、暂缓、降级、关闭候选和下一轮任务生成。Auto-Gate 不把系统内通过等同于投稿级强结论。

自动裁决结果使用：

- `自动通过`
- `自动有条件通过`
- `自动暂缓`
- `自动降级`
- `自动否决`
- `关闭候选`
- `系统内解除阻断候选`

人类确认需求使用：

- `无`
- `工具安装`
- `算力投入`
- `方向偏好`
- `投稿前`
- `公开发布前`
- `强 claim 写入前`

Auto-Gate 的保守默认规则：

- 证据不足时不升级 claim。
- 文献边界不清时不写原创强贡献。
- 实验覆盖不足时不写正式结论。
- 证明缺口存在时继续保留 RD/NXT。
- `claim阻断` 默认继续阻断强 claim；证据链基本齐备时只能先标记为 `关闭候选` 或 `系统内解除阻断候选`。
- 工具安装、算力投入、方向偏好、投稿、公开发布和最终强 claim 表述必须进入人类确认。

关键裁决必须记录：

| 字段 | 内容 |
|---|---|
| Auto-Gate 裁决状态 | 自动通过 / 自动有条件通过 / 自动暂缓 / 自动降级 / 自动否决 / 关闭候选 / 系统内解除阻断候选 |
| Auto-Gate 裁决对象 | claim / RD / NXT / 产物 / 记忆写回 / 角色变更 |
| Auto-Gate 裁决依据 | RD-ID / NXT-ID / EVO-ID / 命题卡 / 文献边界 / 实验日志 |
| 参与裁决角色 | 角色列表 |
| 否决票 | 无 / 角色 + 原因 |
| 保守默认动作 | 不升级 claim / 不关闭债务 / 生成下一轮 NXT / 自动降级 |
| 人类确认需求 | 无 / 工具安装 / 算力投入 / 方向偏好 / 投稿前 / 公开发布前 / 强 claim 写入前 |

## 6. 研究债务驱动下一轮

研究债务不仅是记录，也是下一轮任务包的重要来源。

- `普通` 债务必须持续跟踪，但不阻断本轮继续推进。
- `阻塞` 债务必须生成下一轮处理动作，或说明暂缓理由和负责角色。
- `claim阻断` 债务必须生成下一轮处理动作，并由 Auto-Gate 标记为 `继续阻断`、`关闭候选`、`系统内解除阻断候选` 或 `自动降级`；未解除前不得升级论文强 claim。
- 如果债务暂缓，必须说明暂缓原因、重新触发条件和 Auto-Gate 裁决状态。

`研究债务协调员` 可以汇总、催办和生成下一轮任务包建议，但不能绕过 Auto-Gate 自行关闭 `阻塞` 或 `claim阻断` 债务。

## 7. 研究记忆写回规则

研究记忆只写入跨周期稳定经验，不替代研究债务、claim 审批、文献边界表、Auto-Gate 门禁裁决或最终对外责任确认。

可以写回：

- 可复用证明策略、失败证明路线和常见缺口。
- 可复用实验配置、求解器经验、异常排查经验和复现实验线索。
- 文献边界经验、已有结果边界和 related work 风险模式。
- 方向取舍经验、暂缓方向、失败方向和重新激活条件。

不得写回为稳定结论：

- 未审查 claim。
- 未关闭研究债务。
- 猜想、直觉解释或一次性实验异常。
- 尚未完成文献边界审查的新颖性判断。
- 未经 Auto-Gate 裁决或人类责任确认的主线变化。

写回建议应在 evolution report 中列明写入位置：`方向记忆.md`、`证明策略记忆.md`、`实验记忆.md` 或“无”。

关键任务的 evolution report 还必须填写 EvoScientist 轻量记忆判定字段：

| 字段 | 内容 |
|---|---|
| 是否触发 IDE | 是 / 否；若是，写入方向记忆的候选条目 |
| 是否触发 IVE | 是 / 否；若是，失败分类为 `实现失败 / 证明缺口 / 基础方向失败 / 待判定` |
| 是否触发 ESE | 是 / 否；若是，写入证明策略记忆或实验记忆的候选条目 |
| 是否只是研究债务 | 是 / 否；若是，只进入 RD/NXT，不写成稳定记忆 |
| 证据来源 | EVO-ID / RD-ID / NXT-ID / 命题卡 / 文献边界 / 实验日志 |
| 人类确认边界 | 无 / 工具安装 / 算力投入 / 方向偏好 / 投稿前 / 公开发布前 / 强 claim 写入前 / 不得升级 claim |

若分类为 `待判定`，或涉及未关闭研究债务、未审查 claim、一次性异常，则只能进入 `研究债务队列.md` 或 `下一轮任务包队列.md`，不得写成稳定研究记忆。

## 8. T7 触发规则

出现以下任一情况时，必须判断是否触发 T7：

- 连续两轮出现同类角色缺口。
- 同一角色交付物连续两轮被退回。
- 债务类型为 `待更新角色`。
- 文献边界、证明缺口或实验异常反复阻断同一类任务。
- 当前角色组合无法覆盖新增任务类型或外部输入要求。
- Auto-Gate 或用户要求调整主线、弱化 claim、暂停方向或新增角色。

触发 T7 不等于角色结构自动变化。角色变更必须写入 `角色变更记录.md`，经 Auto-Gate 预审；涉及重大方向、公开责任或强 claim 表述时，再进入人类责任确认。

```markdown
| 角色变更建议 | 无 / RCP-YYYYMMDD-序号 |

```

每轮任务结束必须生成 evolution report，或明确说明“本轮为轻量问答，不产生 report；无新增债务；无下一轮任务包”。涉及关键任务时不得省略 report。

关键任务的 evolution report 必须归档到 `evolution_reports/`，命名与 `轮次 ID` 保持一致。evolution report 只记录本轮闭环状态，不替代 claim 审批、债务关闭、角色变更、Auto-Gate 门禁裁决或最终对外责任确认。

## 11. 兼容旧字段说明

历史文件中的 `PI待裁决事项`、`PI裁决条件`、`关闭裁决人`、`PI裁决需求` 不强制回写重构；自 NXT-20260502-008 起，旧字段默认按以下方式解释：

- 日常推进语境下，等价于 `Auto-Gate 裁决条件`。
- 涉及工具安装、算力投入、方向偏好、投稿、公开发布或强 claim 写入时，等价于 `人类确认需求`。
- 历史记录中的 `PI导师` 不表示用户需要亲自判断每个数学门禁。

## 12. 最小运行规则

- 每轮必须分类：至少说明主任务类型、分类理由、论文锚点和是否关键任务。
- 每轮必须收口：至少说明产物、验收结论、债务影响、Auto-Gate 裁决状态和人类确认需求。
- 每轮必须生成下一轮任务包，或明确“暂无下一轮任务包”。
- 关键任务必须生成 evolution report。
- 关键任务的 evolution report 必须归档，下一轮任务包必须进入队列。
- C-010/C-011 只是当前优先 claim 示例，不是闭环入口；新 claim、新理论方向、新实验任务和新文献边界任务均按 T0-T7 进入闭环。
